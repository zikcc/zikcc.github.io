🔧 一、Linux驱动 - 设备树与驱动匹配

题目1：简述设备树（DTS）在Linux驱动中的作用，并写出匹配驱动的关键数据结构。
答案：

  • 作用：设备树以树状结构描述硬件配置（如寄存器地址、中断号），实现驱动与硬件解耦，避免内核重新编译。
  • 匹配数据结构：
      static const struct of_device_id my_driver_match[] = {
      { .compatible = "vendor,my-device" }, // 设备树中定义的compatible属性
      {}
  };
  MODULE_DEVICE_TABLE(of, my_driver_match);
  ```  
**通俗解释**：  
> 设备树像“硬件说明书”，驱动通过`compatible`字段匹配说明书中的设备型号，实现即插即用。  
**应用场景**：  
嵌入式SoC开发（如TI AM335x、NXP i.MX系列），适配不同硬件版本无需修改驱动代码。

---

### ⚡ 二、中断处理 - 上半部与下半部  
**题目2**：为何Linux中断处理要分上半部（Top Half）和下半部（Bottom Half）？下半部常用哪些实现方式？  
**答案**：  
- **原因**：上半部快速响应硬件中断（关中断执行），下半部处理耗时操作（如数据处理），避免阻塞其他中断。  
- **下半部实现**：  
  - 软中断（SoftIRQ）：内核静态定义，高优先级  
  - Tasklet：基于软中断，可动态创建（原子操作）  
  - 工作队列（Workqueue）：进程上下文，可睡眠  
**技术解析**：  
```c
// 上半部：触发下半部Tasklet
static void my_irq_top_half(struct pt_regs *regs) {
    tasklet_schedule(&my_tasklet); // 调度Tasklet
}

// 下半部：实际处理
static void my_irq_bottom_half(unsigned long data) {
    process_data(); // 可包含复杂逻辑
}
```  
**典型应用**：  
网络设备收包（上半部存数据，下半部协议解析）。

---

### 📡 三、通信协议 - SPI与I2C对比  
**题目3**：列举SPI与I2C的3点核心区别及适用场景。  
**答案**：  
| **特性**       | SPI                          | I2C                  |  
|----------------|------------------------------|----------------------|  
| **信号线**     | 4线（SCK/MOSI/MISO/SS）      | 2线（SDA/SCL）       |  
| **速度**       | 高速（50MHz+）               | 低速（≤400KHz）      |  
| **拓扑**       | 主从+片选（多设备需多SS线）  | 多主多从+地址寻址    |  
**应用场景**：  
- **SPI**：高速Flash读写、TFT屏幕刷新  
- **I2C**：温度传感器（如LM75）、RTC时钟。  

---

### 🔄 四、RTOS任务同步  
**题目4**：FreeRTOS中二值信号量 vs 互斥量的核心区别？互斥量如何解决优先级反转？  
**答案**：  
- **区别**：  
  | **类型**       | 特性                  |  
  |----------------|-----------------------|  
  | 二值信号量     | 无优先级继承          |  
  | 互斥量         | 支持优先级继承        |  
- **优先级反转解决**：  
  低优先级任务占用资源时，临时提升其优先级至高优先级任务水平，防止中优先级任务抢占。  
**代码验证**：  
```c
SemaphoreHandle_t xMutex = xSemaphoreCreateMutex(); // 默认启用优先级继承
```  
**应用场景**：  
SPI总线共享（高优先级任务需快速访问总线时触发继承）。

---

### 💾 五、内核内存管理  
**题目5**：`kmalloc()`与`vmalloc()`的区别？DMA场景应选哪个？  
**答案**：  
- **区别**：  
  - `kmalloc`：分配**物理连续**内存，大小受限（通常≤4MB），访问快  
  - `vmalloc`：分配**虚拟连续**内存（物理可能不连续），支持大内存，访问慢（需页表转换）  
- **DMA选择**：`kmalloc`或`dma_alloc_coherent()`（DMA要求物理连续）。  
**技术解析**：  
```c
// DMA内存申请（物理连续）
dma_addr_t dma_handle;
void *buf = dma_alloc_coherent(dev, size, &dma_handle, GFP_KERNEL);
```  
**典型错误**：  
摄像头数据采集若用`vmalloc`，DMA传输可能因物理地址不连续失败。

### 🤖 六、AI边缘部署优化  
**题目6**：边缘部署千亿模型时，模型压缩技术（量化/蒸馏/剪枝）如何选择？  
**答案**：  
| **技术** | 适用场景                  | 边缘设备约束        |  
|----------|--------------------------|---------------------|  
| 量化     | 推理加速（INT8降内存3x） | 支持INT8的NPU/GPU   |  
| 蒸馏     | 高精度要求（小模型仿大模型） | 训练资源充足       |  
| 剪枝     | 计算资源紧张（稀疏计算） | 支持稀疏加速的硬件 |  
**优化案例**：  
LLaMA-65B部署：  
- **方案**：INT8量化 + 知识蒸馏 → 显存从120GB→24GB  
- **硬件**：NVIDIA Jetson AGX Orin，延迟<100ms。
