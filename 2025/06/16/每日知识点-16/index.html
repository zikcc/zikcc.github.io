题目1：原子操作优化

类型​：性能优化题
​题目​：以下计数器哪种实现性能最优？

// 方案A：std::mutex + int
// 方案B：std::atomic<int>
// 方案C：volatile int

答案​：方案B（std::atomic）
​通俗解释​：原子操作无需锁，CPU指令级保证安全

技术解析​：

std::atomic<int> counter(0);
counter.fetch_add(1, std::memory_order_relaxed); // 无锁高性能

应用场景​：高并发计数器（如QPS统计）

解析：
在多个线程同时修改同一个计数器时（比如高并发网站的访问量统计），如何设计计数器才能既保证正确性又兼顾性能？

// 方案A：std::mutex + int       (互斥锁保护)
// 方案B：std::atomic<int>       (原子操作)
// 方案C：volatile int           ("伪"同步)

方案C：volatile int（危险！）​

volatile int counter = 0;
counter++; // 多线程下不安全

工作原理​：
volatile仅保证变量不被编译器优化（每次从内存读取），​不保证操作的原子性
实际执行counter++包含3步：读值→计算→写回
性能​：🟢 执行快（无同步开销）
​正确性​：❌ ​完全错误​（数据竞争）
​结论​：​绝对禁止用于多线程计数器

方案A：std::mutex + int（安全但性能差）

std::mutex mtx;
int counter = 0;

void increment() {
    std::lock_guard<std::mutex> lock(mtx);
    counter++; // 线程安全但慢
}

工作原理​：
每次操作需要：加锁→修改→解锁
并发时线程会排队等待锁

性能​：
场景 性能表现
低竞争 ⚠️ 一般（约50ns/次）
高竞争 🔴 急剧下降（可能>1000ns/次）
CPU缓存影响 ❌ 缓存失效频繁

方案B：std::atomic（性能最优解）

std::atomic<int> counter(0);
counter.fetch_add(1, std::memory_order_relaxed);

工作原理​：
使用CPU原子指令（如x86的LOCK XADD）
硬件级保证操作不可分割

注意事项​：
使用memory_order_relaxed获得最佳性能（仅保证原子性，无全局同步）
对于简单计数器够用，但需根据场景谨慎选择内存序

扩展知识：
为什么atomic比mutex快这么多？​​

​硬件支持​：直接使用CPU的原子指令，无需系统调用
​无阻塞设计​：CAS（Compare-And-Swap）循环避免线程挂起
​缓存优化​：MESI协议保证缓存一致性，减少总线锁

std::atomic<int> x, y;

// 线程1
x.store(1, std::memory_order_relaxed);  // (1)
y.store(1, std::memory_order_relaxed);  // (2)

// 线程2
while (y.load(std::memory_order_relaxed) != 1) {}  // (3)
std::cout << x.load(std::memory_order_relaxed);    // (4)

可能输出​：0（即使(1)在(2)前执行，线程2也可能先看到y=1而x仍为0）
​若改用memory_order_seq_cst​：保证(1)→(2)和(3)→(4)的顺序，输出必为1。

std::atomic<int> x(0);

// 线程A
x.store(42, std::memory_order_release); // 写操作同步点

// 线程B
std::cout << x.load(std::memory_order_acquire); // 读操作同步点，必输出42
场景 推荐内存序 原因
独立计数器 relaxed 只需原子性，无需同步其他操作
生产者-消费者模型 release(写)+acquire(读) 保证数据发布的可见性
多变量强一致性（如金融交易） seq_cst 避免任何重排，牺牲性能保安全

写同步​：用release保证之前的操作对别人可见。
​读同步​：用acquire保证读到最新数据。
​配对使用​：release和acquire必须成对出现才能生效！

#include <iostream>
#include <atomic>
#include <thread>

std::atomic<int> x(0);
std::atomic<bool> ready(false);

void writer() {
    x.store(42, std::memory_order_release);  // 关键点！
    ready.store(true, std::memory_order_relaxed);
}

void reader() {
    while (!ready.load(std::memory_order_relaxed)) {}
    std::cout << x.load(std::memory_order_acquire); // 必输出42
}

int main() {
    std::thread t1(writer);
    std::thread t2(reader);
    t1.join(); t2.join();
    return 0;
}

输出​：42（若去掉release/acquire，可能输出0）

题目2：死锁预防

类型​：简答题
​题目​：写出两种预防死锁的方法
答案​：

​固定加锁顺序​：所有线程按相同顺序获取锁
​RAII锁管理器​：用std::scoped_lock同时锁定多个互斥量

std::scoped_lock lock(mutex1, mutex2); // C++17

应用场景​：银行转账需锁两个账户时

题目4：生产者-消费者模型

类型​：设计题
​题目​：用条件变量实现生产者-消费者队列，写出伪代码。

std::queue<Data> buffer;
std::mutex mtx;
std::condition_variable cv_producer, cv_consumer;

// 生产者
void producer() {
    std::unique_lock lock(mtx);
    cv_producer.wait(lock, []{ return buffer.size() < MAX_SIZE; });
    buffer.push(data);
    cv_consumer.notify_one();
}

// 消费者（类似，条件取反）

关键点​：双条件变量避免虚假唤醒

题目5：线程局部存储

类型​：选择题
​题目​：以下哪个实现线程独立的计数器？
A. static int count;
B. thread_local int count;
C. std::atomic count;
​答案​：B
​通俗解释​：thread_local为每个线程创建变量副本。
​应用场景​：线程内日志ID、数据库连接

题目6：async vs thread

类型​：简答题
​题目​：std::async和直接创建线程的优势？
​答案​：

async：自动任务调度、可获取返回值（future）、可能复用线程
thread：更精细控制（如分离、优先级）

​应用场景​：需返回值的异步任务用async

题目7：线程安全单例

类型​：代码题
​题目​：写出C++11线程安全的单例模式。

class Singleton {
public:
    static Singleton& get() {
        static Singleton instance; // C++11保证初始化原子性
        return instance;
    }
    Singleton(const Singleton&) = delete;
private:
    Singleton() = default;
};

技术解析​：静态局部变量在C++11后是线程安全的
