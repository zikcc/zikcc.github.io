1. C语言 - 内存安全

  • 类型：代码改错题

  • ​题目​​：以下代码存在什么隐患？如何修正？

    #include <stdio.h>
#include <stdlib.h>

void load_data(int **buf) {
    *buf = (int*)malloc(5 * sizeof(int));
    for (int i = 0; i < 5; i++) {
        (*buf)[i] = i * 10; 
    }
}

int main() {
    int *data = NULL;
    load_data(&data);
    // 使用data...
    free(data);  // 释放内存
    printf("%d", data[0]);  // 问题点
    return 0;
}

  • 答案与解析：

  • 隐患：释放后未置空指针 → 产生悬空指针（Dangling Pointer）

  • 修正方案：

    free(data);
data = NULL;  // 关键修复：释放后立即置空
if (data != NULL) printf("%d", data[0]);  // 安全判断

  • 通俗解释：
    free(data) 如同销毁保险箱，但 data 仍记录着保险箱位置（旧地址）。后续操作如同用废钥匙开锁（data[0]），可能触发崩溃或数据错乱。置为 NULL 相当于丢弃废钥匙。

  • 技术要点：

    // 正确内存生命周期管理：
int *p = malloc(...);   // 分配
if (p != NULL) {...}    // 使用前校验
free(p);                // 释放
p = NULL;               // 置空防误用

  • 典型错误对比：

    错误操作 后果 修正方案
    free 后未置空 悬空指针 → 段错误 p = NULL
    malloc 后漏检查 分配失败导致崩溃 if (p == NULL) exit(1);

2. 数据结构 - 链表实战

  • 类型：编程题

  • ​题目​​：实现​​删除链表中重复节点​​的函数，保留首次出现的节点：

    typedef struct Node {
    int data;
    struct Node *next;
} Node;

void remove_duplicates(Node *head);

  • 参考答案：

    void remove_duplicates(Node *head) {
    Node *current = head;
    while (current != NULL) {
        Node *runner = current;
        while (runner->next != NULL) {
            if (runner->next->data == current->data) {
                Node *temp = runner->next;
                runner->next = runner->next->next;
                free(temp);  // 释放重复节点
            } else {
                runner = runner->next;
            }
        }
        current = current->next;
    }
}

  • 通俗解释：
    想象排查一队人：

    1. 固定当前人（current），派助手（runner）检查后续队伍
    2. 发现重复姓名（相同data）→ 助手移除重复者并记录
    3. 当前人前进，继续排查

  • 技术解析：

    • 双指针技巧：current 锁定目标，runner 扫描后续节点
    • 内存安全：free 后无需置空（局部指针自动失效）
    • 边界处理：头节点无需特殊处理（无前驱节点问题）

  • 应用场景：

    • 数据库去重（如用户ID列表）
    • 编译器符号表处理（避免重复变量名）

3. 网络编程 - TCP协议

  • 类型：简答题

  • ​题目​​：​​TCP三次握手​​中，为什么第二次握手需要同时发送 SYN+ACK？

  • 答案：
    ​​根本原因​​：​​合并确认与同步​​，减少通信轮次，提高效率

    握手过程详解：

    步骤 方向 标志位 作用
    ① 首次握手 客户端→服务端 SYN=1 请求连接 + 发送初始序列号（ISN）
    ② 二次握手 服务端→客户端 SYN=1, ACK=1 确认客户端ISN + 发送自身ISN
    ③ 三次握手 客户端→服务端 ACK=1 确认服务端ISN

  • 为什么不能拆分成两次？

    • 若分开发送（先ACK再SYN）：需4次通信 → 延迟更高
    • SYN+ACK 合并：既确认客户端请求，又声明自身参数 → 最小化建立连接的耗时

  • 底层原理：

    // Linux内核简化实现（syn_sent状态处理）
if (收到SYN) {
    发送SYN+ACK;  // 合并响应
    进入SYN_RECEIVED状态;
}

  • 典型应用：

    • HTTP请求（浏览器访问网页必经历握手）
    • 数据库连接（如MySQL客户端/服务端通信）
