​题目1：线程参数传递

​类型​：代码分析题
题目​：以下代码输出结果是什么？为什么？

void update(int& x) { x *= 2; }
int main() {
    int val = 5;
    std::thread t(update, val);
    t.join();
    std::cout << val; 
}

答案​：输出 5
通俗解释​：
线程默认按值传递参数，val被复制到线程内，原变量未修改。
技术解析​：

// 正确写法：使用std::ref传递引用
std::thread t(update, std::ref(val)); // 输出10

应用场景​：多线程修改共享配置时需传递引用

题目2：互斥锁RAII管理

类型​：简答题
​题目​：std::lock_guard和std::unique_lock的区别？

答案​：

lock_guard：构造时加锁，析构时解锁，不可手动控制
unique_lock：支持延迟加锁、条件变量、锁所有权转移

​通俗解释​：lock_guard像自动门锁，unique_lock像带遥控的手动锁。
​应用场景​：简单同步用lock_guard；条件变量必须用unique_lock

解答：
核心目标相同：​​ 它们俩都像“智能门卫”，帮你自动管理锁（std::mutex）。你开门（加锁）进去办事（访问共享数据），出来时“门卫”自动帮你锁门（解锁），这样你就不会忘记锁门（死锁）了

主要区别在于“灵活性”：​

  1. ​std::lock_guard：简单可靠的“自动门卫”​
  • 工作方式：​​
    你一把它请来（构造它），它立刻就把门锁上（自动加锁）。你办完事，走出办公室（作用域结束），它立刻就把门解锁（自动解锁）
  • 特点：​​
    ​简单直接：​​ 不需要你操心什么时候锁、什么时候解锁。
    ​不能中途开门：​​ 一旦它锁上门，在它“退休”（析构）之前，你不能手动命令它临时开门（解锁），它也不会自己开门
    ​不能迟到：​​ 它一上岗就锁门，你不能让它先等等再锁
    ​不能换岗：​​ 你不能把这个“门卫”转交给别人管理（不可移动）
  • 适用场景：​​
    你进办公室做的事情很简单、很连贯，一口气做完就出来。比如：
void updateSharedCounter() {
    std::lock_guard<std::mutex> guard(shared_mutex); // 进门就锁门
    shared_counter++; // 快速改个数字
    // ... 其他简单操作 ...
} // guard 退休，自动开门
  1. ​std::unique_lock：功能强大的“智能遥控门卫”​
  • 工作方式：​​
    你可以告诉它不同的上岗方式（通过第二个参数）。默认情况下，它和 lock_guard 一样，一来就锁门。但你可以让它更灵活

延迟上岗锁门 (std::defer_lock)：
​​它先来站岗，但不立刻锁门。你可以在需要的时候再命令它锁门（.lock()）。

​接手已锁的门 (std::adopt_lock)：​​
如果门已经被你手动锁上了，你可以告诉它来接手，它只负责最后帮你解锁。

​尝试锁门 (std::try_to_lock)：​​
它去试试锁门，如果锁被别人占着锁不上，它不会傻等，会告诉你“没锁上”，你可以干点别的。

  • 特点：​​
    ​灵活控制：​​
    你可以随时命令它锁门（.lock()）或者临时开门（.unlock()）​。开了门你可以在外面办点不涉及共享资源的事，办完再命令它锁门（.lock()）你继续进去干共享资源的活。

​可以换岗：​​
你可以把这个“门卫”的管理权转交给另一个地方（支持移动语义）。
​
能和“通知员”（条件变量）配合：​​
这是它最重要的特殊技能！当“通知员”（std::condition_variable）需要你在等待时临时开门让其他线程进来干活时，必须用 unique_lock。lock_guard 做不到这点。

  • 适用场景：​​
    ​场景一：​​ 你需要先做些不涉及共享数据的准备工作，准备好了再进去操作共享数据
void complexTask() {
    // 先做一些不涉及共享数据的准备工作...
    std::unique_lock<std::mutex> ulock(shared_mutex, std::defer_lock); // 门卫来了，但门没锁
    // ... 继续做准备工作 ...
    ulock.lock(); // 准备工作做完，现在命令门卫锁门，开始操作共享数据
    // ... 操作共享数据 ...
    ulock.unlock(); // 共享数据操作完了，临时开门，出去做点收尾工作（不涉共享）
    // ... 收尾工作 ...
} // ulock 退休，如果门开着它会自动锁上

场景二：​​ ​配合“通知员”（条件变量）等待某个条件成立

std::mutex mtx;
std::condition_variable cv;
bool data_ready = false;

void consumer() {
    std::unique_lock<std::mutex> ulock(mtx); // 锁门进去检查
    while (!data_ready) { // 条件没满足
        cv.wait(ulock); // 关键！通知员会命令门卫临时开门(解锁)并让你睡觉。醒来后门卫自动重新锁门。
    }
    // 此时 data_ready 为 true，且门是锁着的，安全消费数据...
}

场景三：​​ 你需要临时释放锁去做别的事情。

​场景四：​​ 你需要转移锁的所有权​（较少见）。

总结对比表：​

std::lock_guard (自动门卫) std::unique_lock (遥控门卫)
自动加锁解锁(核心) ✅ 构造时加锁，析构时解锁 ✅ 构造时可选择加锁，析构时解锁
能否手动解锁? ❌ 不行 ✅ 可以 .unlock()
能否手动加锁? ❌ 不行 ✅ 可以 .lock() (如果构造时没加)
能否延迟加锁? ❌ 不行 ✅ 可以 (std::defer_lock)
能否接管已加锁的互斥量? ❌ 不行 ✅ 可以 (std::adopt_lock)
能否尝试加锁? ❌ 不行 ✅ 可以 (std::try_to_lock)
能否转移所有权? ❌ 不行 ✅ 可以 (移动语义)
能否配合条件变量? ❌ 不行 ✅ 必须用它
性能开销 ⭐⭐ 更低 (简单) ⭐ 更高 (功能多，需维护状态)
典型使用场景 简单、快速的临界区操作 需要灵活控制锁、配合条件变量、复杂同步逻辑

通俗解释（用门卫比喻）：​​

std::lock_guard 就像一个非常尽责但死板的门卫。你一把钥匙（互斥量）交给他，他立刻就把门锁上。你进门办事，办完了出来，他立刻又把门锁上。整个过程你不能命令他“先别锁”、“现在开一下门我出去透透气”。他就是锁门->开门->锁门，简单可靠。

std::unique_lock 就像一个更智能、听指挥的门卫。你可以告诉他：“拿着钥匙，但先别锁门”（defer_lock），等你准备好了喊一声“锁门”他才锁（lock()）。办完共享数据的事，你可以说“开门，我出去办点私事”（unlock()），他开门让你出去，你办完私事回来喊“锁门”（lock()）他又锁上门让你继续操作共享数据。最重要的是，它能和“通知员”（条件变量）完美配合，在需要等待时帮你开门并让你休息。

选择建议：​​

​能用 lock_guard 就用它：​​
它简单高效，适用于绝大多数只需要在作用域内保护共享数据的场景。
​需要灵活性就用 unique_lock：​​
当你需要延迟加锁、手动解锁、尝试加锁、配合条件变量或者转移锁的所有权时，​必须使用 unique_lock。特别是条件变量，只能用 unique_lock​

题目3：条件变量陷阱

类型​：代码改错题
​题目​：修复以下代码的死锁问题：

std::mutex mtx;
std::condition_variable cv;
bool ready = false;

void consumer() {
    std::lock_guard<std::mutex> lock(mtx); // 错误点
    cv.wait(lock, []{ return ready; });    // 死锁！
}

答案：

void consumer() {
    std::unique_lock<std::mutex> lock(mtx); // 必须用unique_lock
    cv.wait(lock, []{ return ready; });
}

技术解析​：condition_variable.wait() 只接受unique_lock，因其需临时解锁

wait() 的内部工作流程:
wait() 的操作是原子化的，包含三个关键步骤：

  • 检查条件并解锁​：
    线程调用 wait() 时，首先检查条件（若提供 Predicate 参数）。
    条件满足时​：不释放锁、不阻塞，直接继续执行
    条件不满足时​：原子化执行解锁+阻塞​（非先解锁后阻塞）
    解锁+阻塞必须是原子操作​：这是标准强制要求，具体实现可能：
// 伪代码：Linux futex 实现
void unlock_wait_lock() {
    atomic {
        mutex.unlock();
        add_to_wait_queue();  // 原子操作内完成解锁和加入等待队列
    }
    sleep(); // 阻塞
}

  • ​阻塞等待通知​：
    线程进入阻塞状态，直到被 notify_one() 或 notify_all() 唤醒，或遭遇虚假唤醒。

  • ​唤醒后重新加锁​：
    线程被唤醒时，​自动重新获取互斥锁​（lck.lock()），恢复对共享数据的独占访问
