每日知识点-21

今日学习内容聚焦于Linux内核机制与系统调用，涵盖进程调度、虚拟内存管理、文件系统VFS及epoll机制。

今日学习任务：Linux内核核心机制
模块概览：

  1. 进程调度原理：CFS调度器、实时进程调度
  2. 虚拟内存管理：分页机制、伙伴系统
  3. 文件系统VFS：inode、dentry、文件读写流程
  4. epoll实现机制：高效I/O多路复用原理

题目1：进程调度 - CFS调度器原理

类型：简答题
题目：Linux的CFS调度器如何保证进程的“完全公平”？vruntime的作用是什么？
参考答案：
CFS通过红黑树管理可运行进程，按vruntime（虚拟运行时间）排序。每次调度选择vruntime最小的进程运行。vruntime的增长速度与进程权重成反比（高优先级进程增长慢），从而保证低优先级进程也能获得CPU时间。

通俗解释：
想象全班同学轮流回答问题，老师优先让发言次数少的同学回答（vruntime小的进程）。成绩好的同学（高权重）被叫到的间隔更长，避免学霸独占课堂时间。

技术解析：

  • 权重计算：nice值映射到权重（nice=0时权重1024，每降低1级权重增加约25%）。
  • 时间片分配：进程运行时间 = 调度周期 × (进程权重 / 所有进程权重之和)。
  • 代码片段：
    // 内核中vruntime更新逻辑（简化）
vruntime += delta_exec * (NICE_0_LOAD / weight); // delta_exec:实际运行时间

应用场景：
Web服务器处理大量并发请求时，CFS避免某个进程长期占用CPU导致其他请求饿死。

题目2：进程调度 - 实时进程优先级

类型：编程题
题目：编写C程序，创建一个实时进程（SCHED_FIFO），优先级为50，循环打印消息。观察其是否阻塞普通进程。
参考答案：

#include <sched.h>
#include <stdio.h>
#include <unistd.h>

int main() {
    struct sched_param param = {.sched_priority = 50};
    sched_setscheduler(0, SCHED_FIFO, &param);  // 设置实时策略

    while (1) {
        printf("Real-time process running\n");
        sleep(1);
    }
    return 0;
}

编译命令：gcc -o realtime realtime.c
运行需root权限：sudo ./realtime

通俗解释：
实时进程像急诊医生，优先级高于普通门诊（普通进程）。一旦启动，除非主动休息或更高优先级急诊插入，否则持续工作。

技术解析：

  • SCHED_FIFO：无时间片概念，一直运行直到主动放弃或被更高优先级抢占。
  • 风险：若循环中无阻塞调用（如sleep），普通进程将完全无法执行（系统可能卡死）。

应用场景：工业机器人控制、音视频流处理等实时性要求高的场景。

题目3：虚拟内存管理 - 分页机制

类型：选择题
题目：Linux中32位系统采用二级页表，64位系统通常用几级页表？
A. 2级
B. 3级
C. 4级
D. 5级

答案：C（x86_64使用4级页表：PGD→PUD→PMD→PT）。

通俗解释：
内存地址像快递地址：国家→省→市→区→门牌号。64位地址空间大，需更多层级才能快速定位。

技术解析：

  • 页表作用：将虚拟地址映射到物理地址。
  • 性能优化：TLB缓存常用映射，减少查表次数。

题目4：虚拟内存管理 - 伙伴系统

类型：简答题
题目：伙伴系统（Buddy System）解决什么问题？如何分配和释放内存？
参考答案：
问题：减少内存碎片，快速分配连续物理页。
分配流程：

  1. 请求2^k大小内存，从2^k链表查找空闲块。
  2. 若无空闲，拆分更大块（如2^{k+1}）为两个2^k“伙伴”，一个分配，一个挂入链表。

释放流程：释放后检查相邻伙伴是否空闲，是则合并成更大块。

通俗解释：
像拆快递箱：需要小箱时拆大箱（拆分），用完的小箱和邻居空箱合并回大箱（合并）。

题目5：文件系统VFS - inode与dentry

类型：简答题
题目：解释VFS中inode和dentry的作用及区别。
参考答案：

  • inode：存储文件元数据（权限、大小、块位置），唯一标识文件。
  • dentry：目录项，缓存路径到inode的映射（如/home/a.txt），加速路径查找。

通俗解释：

  • inode：身份证（唯一身份信息）。
  • dentry：通讯录条目（名字到身份证号的映射）。

技术解析：

// 内核inode结构（简化）
struct inode {
    umode_t i_mode;     // 文件类型和权限
    loff_t i_size;      // 文件大小
    struct address_space *i_mapping; // 数据块映射
};

题目6：文件系统VFS - 文件读写流程

类型：编程题
题目：编写程序使用open和read系统调用读取文件，并打印内容。说明内核中从系统调用到磁盘读数据的流程。
参考答案：

#include <fcntl.h>
#include <stdio.h>
#include <unistd.h>

int main() {
    char buf[1024];
    int fd = open("test.txt", O_RDONLY);
    ssize_t n = read(fd, buf, sizeof(buf));
    write(STDOUT_FILENO, buf, n);
    close(fd);
    return 0;
}

内核流程：

  1. read() → sys_read() → VFS层找到文件inode。
  2. 调用文件系统具体函数（如ext4_read_inode()）。
  3. 通过页缓存读取数据，若未命中则触发磁盘I/O。

题目7：epoll机制 - 边缘触发（ET）与水平触发（LT）

类型：选择题
题目：关于epoll的触发模式，以下说法正确的是？
A. ET模式需一次性处理完所有事件，否则可能丢失事件
B. LT模式会重复通知未处理事件
C. ET模式效率更高，但编程更复杂
D. 以上全部

答案：D 。

通俗解释：

  • LT模式：快递短信提醒（一直提醒直到取件）。
  • ET模式：单次门铃（按一次后不重复，需一次性搬完所有快递）。

题目8：epoll机制 - 高效性原理

类型：简答题
题目：为什么epoll比select/poll更高效？
参考答案：

  1. 无需遍历所有fd：epoll仅返回就绪的fd，而select/poll需遍历全部fd。
  2. 内核事件表：epoll_ctl注册fd时，内核会监听其事件，无需每次重新传入fd集合。

应用场景：10万+并发连接的Web服务器（如Nginx）。

题目9：epoll编程实战

类型：编程题
题目：编写epoll程序，监听标准输入，ET模式下读取数据并打印。
参考答案：

#include <sys/epoll.h>
#include <unistd.h>
#include <stdio.h>

int main() {
    int epfd = epoll_create1(0);
    struct epoll_event ev, events[10];
    ev.events = EPOLLIN | EPOLLET;  // 边缘触发
    ev.data.fd = STDIN_FILENO;
    epoll_ctl(epfd, EPOLL_CTL_ADD, STDIN_FILENO, &ev);

    while (1) {
        int n = epoll_wait(epfd, events, 10, -1);
        for (int i = 0; i < n; i++) {
            char buf[1024];
            ssize_t len = read(events[i].data.fd, buf, sizeof(buf));
            write(STDOUT_FILENO, buf, len);
        }
    }
}

题目10：综合应用 - 文件服务器设计

类型：设计题
题目：如何用epoll+多线程实现高并发文件服务器？简述架构。
参考答案：

  1. 主线程：epoll监听连接请求（LT模式）。
  2. 工作线程池：接收连接后，将fd分发给线程（用Round-Robin）。
  3. 线程内：epoll_wait处理已连接fd的读写（ET模式），读取文件后通过sendfile零拷贝发送。

薄弱项：伙伴系统合并流程、ET模式非阻塞编程需加强。
