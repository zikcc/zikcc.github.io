自定义协议栈服务器开发-4

一、ioctl：通用设备控制接口

1. 什么是 ioctl(fd, request, argp)？

  • fd：之前用 open() 得到的设备文件描述符（这里是 /dev/video0）。
  • request：一个数字常量，告诉内核“我要对这个设备做什么操作”。
  • argp：指向一个结构体或整数的指针，存放或接收额外的参数／返回值。

简单类比：

  • read(fd, buf, len)：从设备读数据
  • write(fd, buf, len)：向设备写数据
  • ioctl(fd, XXX, &y)：向设备发送“指令”XXX，可能携带或返回数据y

2. V4L2 常用的几个 ioctl 请求（request）

请求宏 作用
VIDIOC_REQBUFS “请为我申请 N 个缓冲区”（参数是 v4l2_requestbuffers）
VIDIOC_QUERYBUF “告诉我第 i 个缓冲区的大小和在内核空间的偏移量”（参数是 v4l2_buffer）
VIDIOC_QBUF “把编号为 i 的空缓冲区放入内核的采集队列”，让驱动往里写新帧
VIDIOC_DQBUF “从内核队列里取出一个已写满帧的缓冲区”，交给用户进程处理
VIDIOC_STREAMON “启动流模式”，开始不停往缓冲区写入帧
VIDIOC_STREAMOFF “停止流模式”，停止往缓冲区写帧
  • VIDIOC_REQBUFS：告诉驱动“我想用 mmap 模式，一次准备读取 4 帧数据，快给我分配 4 块内核缓冲区”。
  • VIDIOC_QUERYBUF：对每块缓冲区执行此命令后，v4l2_buffer 结构体会告诉你：
    • length：这块缓存的字节大小
    • m.offset：这块缓存相对于设备文件起始处的偏移量
  • VIDIOC_QBUF：把一块空的、已经 QUERYBUF 过的缓存放到“待写入”队列里。驱动拿到后会往里写下一帧。
  • VIDIOC_STREAMON / STREAMOFF：启动或停止“流”模式，让驱动开始/停止循环地往这些队列缓存里写入／停止写入帧。
  • VIDIOC_DQBUF：从“已写入”队列里取出一个缓存，告诉你它是哪块（buf.index），以及写入了多少字节（buf.bytesused），用户层就能处理这帧数据。

二、mmap：内存映射，让用户进程直接访问缓冲区

1. 为什么要用 mmap 而不是 read？

  • read()：每次都得内核→用户空间拷贝一次，效率较低。
  • mmap()：把内核分配的那块物理内存“映射”到用户进程地址空间，同一块内存，内核和用户共用一份，不用额外拷贝，速度更快。

2. mmap 怎么用？

void* addr = mmap(
    nullptr,       // 让系统自动选择一个合适的起始地址
    length,        // 映射大小（字节数）
    PROT_READ      // 用户进程可读
    | PROT_WRITE,  // 用户进程可写（对 MMAP 这种设备来说一般写入无用，但必须有）
    MAP_SHARED,    // 多个进程（内核/用户）共享这块映射，写到它会通知内核
    fd,            // 哪个文件或设备
    offset         // 从文件(设备)的哪个偏移量开始映射
);
  • length：我们之前用 VIDIOC_QUERYBUF 得到过，它告诉这个缓冲区有多大。
  • offset：同样来自 VIDIOC_QUERYBUF，表示内核里这块缓冲区相对于设备文件开始的位置。
  • 返回值 addr：之后你就可以像读普通内存一样读 addr[0]、addr[1]…，读到的就是摄像头给你写的帧数据。

3. 映射后，你需要做两件事：

  1. 入队（VIDIOC_QBUF）——让驱动知道“这块内存我准备好了，快写新帧进来”。
  2. 取队（VIDIOC_DQBUF）——告诉驱动“我处理完了，把写好的帧拿给我”，此时 buf.index 告诉你是哪块，bytesused 告诉你写了多少字节，你就用 mmap 返回的指针去拷贝或解码。

最终，零拷贝、高并发地完成了“生产（驱动写帧）—消费（用户进程读帧）”的循环。
