💡 总结：一句话解释整个流程

主线程只管 epoll + 分发，线程池负责干活（处理数据并回复），所有 socket 都是非阻塞的。

    ╔═════════════════╗
    ║          内核 (TCP/IP栈)     ║

    ║  conn_fd2 (连接 socket) ◄───── 客户端2接入
    ║  conn_fd3 (连接 socket) ◄───── 客户端3接入

                │
        [线程池处理业务逻辑]

🔍 一步步解释结构组成：

1. listen_fd：监听 socket

  • 你用 socket(), bind(), listen() 创建的最初的 socket。
  • 只负责监听端口是否有新的连接进来。
  • 当有客户端来连接，就会触发 epoll 事件，你再用 accept() 把连接拿出来。

2. conn_fd：已连接 socket

  • 每次 accept() 返回一个新的 socket，这就是你和客户端之间通信的 socket。
  • 每个客户端一个连接，一个 fd。
  • 你用 read()/write() 或 recv()/send() 来与该客户端进行数据交换。

3. epoll 监听这些连接 fd

  • epoll 用来监听 conn_fd1, conn_fd2, …, 发生事件（如可读/可写）。
  • 一旦某个 fd 有数据，epoll 告诉你，然后你就用线程池去处理。

4. 线程池做处理，不阻塞主线程

  • 主线程只负责接收连接、触发 epoll。
  • 每当有数据到达，就把这个事件扔进线程池处理。

✅ 为什么这种设计适合高并发？

优势 说明
高性能 epoll 支持百万连接，O(1) 的事件通知
每个客户端独立 fd 无共享，不影响彼此
非阻塞 IO 不卡线程
线程池 避免线程爆炸，任务处理更平滑
可扩展性强 后期可加入定时器、连接池、协程等机制

重构方案概览

模块 原问题 重构方法 代码示例 不改的坏处 改动的好处
协议层 无发送缓冲区导致数据丢失 拆分send_packet为数据入队和实际发送 enqueuePacket() + flushSendBuffer() 数据可能丢失 支持部分发送和重试机制
协议层 无接收缓冲区导致粘包处理错误 引入接收缓冲区 tryReceivePacket() 数据解析失败 正确处理TCP流式数据
连接层 直接操作裸指针导致线程竞争 使用智能指针+互斥锁 std::shared_ptr<Connection> 数据竞争导致崩溃 线程安全读写
事件循环 递归调用handleWrite 通过EPOLLOUT事件触发发送 triggerWriteEvent() 栈溢出风险 解耦读写逻辑
资源管理 手动关闭socket易遗漏 RAII包装文件描述符 ScopedFileDescriptor 资源泄漏 自动释放资源

阻塞模式 vs 非阻塞模式的核心区别

特征 阻塞模式 非阻塞模式
read/write 行为 操作未完成时线程挂起，直到数据就绪或超时 操作立即返回，若数据未就绪返回错误码（EAGAIN 或 EWOULDBLOCK）
适用场景 简单客户端、低并发服务端 高并发服务端（如 Nginx、Redis）
资源占用 每个连接需独占线程，线程数随连接数线性增长 单线程可处理数千连接（通过事件驱动）

阻塞 vs 非阻塞模式的核心区别

socket 模式 read 行为 代码表现
阻塞模式 若内核缓冲区无数据，read 挂起线程，直到数据到达或超时。 线程卡在 read 处，无法处理其他任务。
非阻塞模式 若内核缓冲区无数据，read 立即返回错误，设置 errno 为 EAGAIN 或 EWOULDBLOCK。 线程继续执行，需检查 errno 并重试。

为何必须使用接收缓冲区？

TCP 协议的核心特性

  • 流式协议：无消息边界，数据像水流一样连续。
  • 可能分片：一个应用层数据包可能被拆分为多个TCP段发送。

示例：粘包与半包

  • 无缓冲区：无法正确识别 Packet1 和 Packet2 的边界。
  • 有缓冲区：积累数据，直到能提取完整包。
