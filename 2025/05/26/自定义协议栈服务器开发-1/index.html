自定义协议栈服务器开发-1

文件名 作用描述
Packet.hpp/.cpp 定义数据包结构，序列化和反序列化函数
Protocol.hpp/.cpp 网络通信的封装（send/recv）
Server.cpp TCP服务端，接收客户端消息并回复
Client.cpp TCP客户端，发送消息并读取服务器回应

大厂级高性能 TCP 服务端架构思路及示例代码片段，基于 Reactor 模式 + epoll + 线程池，把网络 I/O 和业务处理彻底拆开，保证主线程专注于 I/O，所有耗时业务（如爬虫、协议解析）都扔给工作线程去做。

✅ 1. 单线程 + 非阻塞 I/O

传统的服务端模型（例如阻塞式 accept-read-write）每个客户端都需要一个线程。假设每个线程消耗 1MB 栈空间，1万连接就需要 10GB 内存，而且线程调度开销极大。

非阻塞 I/O + epoll 只需要一个主线程就可以同时管理上万连接。它不会因为某个连接的数据没准备好而阻塞。

✅ 2. epoll 是 O(1) 的 I/O 多路复用

  • select/poll 每次都要遍历所有 fd，时间复杂度 O(n)。
  • epoll 只返回就绪的文件描述符列表，时间复杂度接近 O(1)。

所以 epoll 对大并发连接场景特别高效。

✅ 3. 边沿触发（Edge Triggered）减少系统调用

epoll 支持两种模式：

  • Level Trigger（水平触发）：数据没读完会一直通知。
  • Edge Trigger（边沿触发）：只在状态变化时通知一次，需要把 buffer 一次读干净。

这显著减少了 epoll_wait() 的返回次数，降低系统调用频率。

✅ 4. 主线程不做业务，线程池异步执行逻辑

主线程只管网络事件，不涉及业务计算（如爬虫、数据库等耗时任务），这样它能持续保持高响应性。

业务线程通过线程池统一调度，充分利用多核 CPU 的并发能力，避免线程频繁创建/销毁的性能浪费。

✅ 5. 内存复用和缓冲池（可拓展优化）

  • 不频繁 new/delete → 对象池（Object Pool）
  • 避免多线程 malloc/free 抢锁 → Arena 分配器
  • 提前准备 buffer，提高 cache 命中率

✅ 6. 异步写回避免阻塞

如果你直接在主线程或业务线程里 write() 客户端，可能因为客户端读太慢导致阻塞（内核 send buffer 满）。

我们这里的做法：

  • 把写数据加入一个 outbuf 中
  • 主线程监听 EPOLLOUT 时间再去写
  • 一次写不完就下次继续

这样即使写出慢，也不会阻塞主线程或业务线程。

✅ 7. 高可扩展性

你可以非常轻松地做出扩展：

  • ✅ 加缓存（如 LRU）
  • ✅ 加限流（如令牌桶）
  • ✅ 加负载均衡（Nginx 前置）
  • ✅ 微服务化部署（爬虫、数据分析、识别模块独立服务）

🔧 总结架构优势（一图看懂）

模块 技术方案 好处
I/O 多路复用 epoll + 非阻塞 I/O 管理上万连接，高性能低开销
事件处理 Reactor 模式 解耦 I/O 与业务，低延迟响应
并发处理 线程池 高并发、稳定、可复用线程资源
写回机制 异步写 + 缓冲区 不阻塞 I/O，不影响主线程性能
扩展性 模块化、插件化 支持微服务、可水平扩展

💬 现代大厂使用案例对比

大厂 使用模式 对应点
Nginx epoll + 非阻塞 + 事件驱动 完整的 Reactor 模式
Redis 单线程 + epoll 利用 epoll 高效调度请求
Netty Reactor + 线程池 + 编解码器 和我们这套结构非常相似
MySQL Proxy 主线程监听连接 + worker 处理查询 主从分离的 I/O 与业务处理思想
