1. C++中引用(&)和指针(*)的区别

  • 通俗解释​​：
    引用像​​身份证复印件​​——必须绑定原身份（初始化后不可改），操作复印件等同操作本人；指针像​​快递单号​​——可随时修改指向的包裹（可重赋值），
    但可能填错单号（空指针）。
  • ​技术解析​​：
    引用：​​不可为空​​、​​不可重绑定​​、​​无独立内存​​
    指针：可为空、可重指向、有独立内存地址 
    int a = 10;
int& ref = a;  // 引用必须初始化，ref是a的别名
int* ptr = &a; // 指针存储地址，可修改 ptr = nullptr;
  • ​​应用场景​​：
    函数参数传递：引用避免拷贝（void print(const string& s)），指针用于需重指向的场景（链表操作）
  • ​​避坑指南​​：
    函数返回局部变量引用会导致悬空引用！应返回静态变量或传值

2. ​​const关键字的作用​

  • 通俗解释​​：
    const是​​防篡改封条​​：贴变量上（值不可改），贴函数参数上（函数内不可改），贴成员函数上（函数不改对象状态）。
  • ​技术解析​​： 
    const int MAX = 100;       // 常量不可修改
void print(const Student& s); // 保证不修改s
class Circle {
double getArea() const;  // 承诺不修改类成员
};
    常量指针：不能通过指针修改指向的值，变量自身不可变（const int* p）
    指针常量：不能改变指针的指向，指向的对象不可变（int* const p）
  • 应用场景​​：
    硬件寄存器映射（volatile const uint32_t* REG_ADDR），防止误写

3. ​​虚函数实现原理​​

  • 通俗解释​​：
    父类定义​​万能遥控器按钮​​（虚函数），子类​​重写按钮功能​​。按父类按钮（基类指针）实际执行子类功能（动态绑定）。
  • 技术解析​​： 
    class Animal {
public:
virtual void speak() { cout << "???"; } // 虚函数 → 生成虚函数表
};
class Dog : public Animal {
void speak() override { cout << "Wang!"; } // 覆盖虚函数表项
};
Animal* a = new Dog();
a->speak(); // 输出 "Wang!" （查虚函数表调用Dog::speak）
    核心机制：​​虚函数表（vtable）​​ + ​​虚表指针（vptr）​​
    代价：每个对象增加8字节（64位）
  • 应用场景​​：
    插件架构（基类定义接口，动态库实现具体功能）

4. 进程 vs 线程的区别

对比项 进程 线程
🍎 通俗比喻 独立办公室（有独立资源） 办公室内同事（共享资源）
资源隔离 内存、文件句柄独立 共享进程内存和文件
创建开销 大（需复制父进程资源） 小（仅复制栈和寄存器）
通信成本 高（需IPC：管道/共享内存） 低（直接读写共享变量）
崩溃影响 不影响其他进程 同一进程内线程全部终止
  • 应用场景：
    Chrome浏览器：每个标签页是独立进程（崩溃隔离），页面渲染用多线程

5. 互斥锁 vs 自旋锁的选择

  • 通俗解释：

    • 互斥锁：发现厕所有人→去休息室睡觉（线程休眠），等人出来再被唤醒
    • 自旋锁：发现厕所有人→门口转圈盯门（CPU空转），门开立即冲入

  • 技术解析：

    锁类型 适用场景 临界区耗时 性能影响
    互斥锁 I/O操作、复杂计算 >1μs 上下文切换开销大
    自旋锁 短时操作（如计数器自增） <1μs 浪费CPU但响应快

  • 应用场景：
    网卡收包中断处理（自旋锁），数据库连接池管理（互斥锁）

6. 虚拟内存的作用

  • 通俗解释：
    虚拟内存像酒店房间管理系统：
    - 每个程序觉得独占所有房间（连续地址空间）
    - 实际房间（物理内存）可能不足，部分客人行李存车库（磁盘交换区）
    - 保安（MMU）实时查房卡（页表）找真实房间号
  • 技术解析：
    • 页表（Page Table）：映射虚拟页→物理页帧
    • 缺页中断：访问未加载页时触发，OS从磁盘调入
    • 写时复制（Copy-on-Write）：子进程共享父进程页，仅修改时复制
  • 应用场景：
    运行超内存程序（如8GB内存启动16GB的VMware），进程隔离（防止A进程篡改B进程数据）

​7. ​哈希表冲突解决方案​

  • 通俗解释​​：
    两辆车（数据）被导航到同一车位（哈希冲突）：
    ​​链地址法​​：车位改成立体车库（链表挂多辆车）
    ​​开放寻址法​​：保安找下一个空位（线性/二次探测）

  • 技术解析​​：

    // 链地址法实现（C++简版）
class HashMap {
vector<list<pair<int, string>>> buckets; // 桶数组+链表
void insert(int key, string val) {
    int idx = hash(key) % buckets.size();
    for (auto& p : buckets[idx]) {
    if (p.first == key) { p.second = val; return; } // 键存在则更新
    }
    buckets[idx].push_back({key, val}); // 冲突时链表追加
}
};
    方法 平均时间复杂度 空间开销 适用场景
    链地址法 O(1 + α) 指针额外空间 通用场景
    开放寻址法 O(1/(1-α)) 无指针开销 缓存友好，内存紧张

  • 应用场景​​：
    Redis字典（链地址法），Linux内核页缓存（开放寻址）

8. 二叉树前序/中序/后序遍历​​

  • 通俗解释​​：
    遍历像​​查户口登记​​：
    ​​前序​​：先登记自己，再查左子树，最后右子树（根→左→右）
    ​​中序​​：先查左子树，再登记自己，最后右子树（左→根→右）→ ​​二叉搜索树输出有序序列​​
    ​​后序​​：先查左右子树，最后登记自己（左→右→根）→ ​​用于释放子树内存​

  • 技术解析​​：

    // 递归中序遍历（升序输出二叉搜索树）
void inorder(TreeNode* root) {
if (!root) return;
inorder(root->left);     // 先左子树
cout << root->val << " "; // 输出当前节点
inorder(root->right);    // 再右子树
}
    遍历方式 非递归实现核心 应用场景
    前序 栈顶→右子入栈→左子入栈 目录树结构展示
    中序 左链入栈→弹出访问→转右子树 二叉搜索树有序输出
    后序 双栈法/反向前序 表达式树求值（后缀表达式）

  • 应用场景​​：
    编译器解析数学表达式（后序遍历求值），文件系统树状展示（前序遍历）

​9. ​TCP为什么需要三次握手？

  • 通俗解释​​：
    类比打电话：
    A打给B：“能听到吗？”（SYN）
    B回答：“能听到，你听得到我吗？”（SYN-ACK）
    A说：“我也能听到”（ACK）→ ​​双方确认通信能力​​
    如果只有两次：B无法确认A是否收到回复（可能B白等）
  • 技术解析​​： 
    # TCP握手抓包示例（Wireshark）
1. Client → Server: [SYN] Seq=0
2. Server → Client: [SYN, ACK] Seq=0, Ack=1
3. Client → Server: [ACK] Seq=1, Ack=1
    ​- ​防历史连接​​：避免旧重复SYN导致错误连接
    ​- ​同步初始序列号​​：防止网络延迟导致数据错乱
  • ​​避坑指南​​：
    服务端未收到第三次ACK会重传SYN-ACK（默认重试5次），可能被SYN Flood攻击！

10. Epoll比Select高效的原因​

  • 通俗解释​​：
    ​​Select​​：快递员每次挨个问1000个货架（fd）：“有包裹吗？”（O(n)遍历）
    ​​Epoll​​：货架装传感器，快递员直接看指示灯（事件驱动），哪亮去哪（O(1)定位）

  • 技术解析​​：

    // Epoll核心API
int epfd = epoll_create1(0);  // 创建epoll实例
struct epoll_event ev;
ev.events = EPOLLIN;          // 监听可读事件
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev); // 注册socket

while (1) {
    int n = epoll_wait(epfd, events, MAX_EVENTS, -1); // 等待事件
    for (int i = 0; i < n; i++) {
        if (events[i].events & EPOLLIN) {
        handle_read(events[i].data.fd); // 直接处理活跃fd
        }
    }
}
    模型 时间复杂度 最大fd数 内核态拷贝
    select O(n) 1024 每次全量拷贝fd集合
    epoll O(1) 10万+ 仅返回活跃fd

  • 应用场景​​：
    Nginx高并发连接（10万+），Redis事件处理
